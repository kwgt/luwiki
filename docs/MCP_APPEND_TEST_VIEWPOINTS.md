# MCP `append` テスト観点一覧

本書は、MCPサーバ機能における `append` の競合制御および amend 挙動のテスト観点を整理した独立したテスト観点書である。

位置付けとしては、`docs/MCP_IMPLEMENTATION_DESIGN_TASKS.md` の 4.9.2 に対応する成果物であり、
MCP 設計書群から参照される実装・テスト実装向け文書として扱う。

`append` は MCP 専用の更新操作であり、
「最新本文末尾への単純追記」「限定条件下での amend 相当保存」
「通常更新より緩い競合制御を許可しない」
「必要に応じた短時間待機と timeout 時の conflict」
を同時に満たす必要がある。
そのため本書では、単なる本文更新テストではなく、
revision 判定、ロック待機、再読込、エラー写像まで含めて観点を整理する。

## 2. 観点の使い方

- 本書は `append` の保存ロジックに対する単体テスト、サービス層結合テスト、DB連動テスト、MCP 統合テストへ観点を配分するための別紙である
- `append` は公開面としては MCP 専用だが、内部では既存保存 API や lock 状態確認と接続するため、公開面だけで閉じた確認にしない
- 本書では「同一ユーザ amend」「別ユーザ時の新規 revision」「ロック待機」「競合 timeout」の 4 本柱を最低限の完了条件とし、その周辺条件も含めて整理する
- 監査ログ集約観点は本書では主対象にせず、4.9.3 側で扱う

## 3. 単純追記の観点

### 3.1 基本意味論

1. `append` が既存本文の末尾へ `content` をそのまま連結すること
2. 既存本文の途中挿入、先頭挿入、置換、削除を行わないこと
3. クライアントが送るのは全文ではなく差分文字列のみであること
4. 空ページに対する `append` が許可されること
5. 空文字追記が `InvalidInput` になること

### 3.2 改行規則

1. 既存本文末尾と追記文字列先頭の間に改行を自動補完しないこと
2. 追記文字列中の改行がサーバ側で書き換えられないこと
3. 行分離が必要な場合はクライアント指定どおりの改行だけが反映されること

## 4. amend 相当判定の観点

### 4.1 同一ユーザ amend

1. 最新 revision の直前更新者と今回の操作ユーザが同一である場合、amend 相当保存になること
2. amend 相当保存時に結果 `revision` が既存最新 revision を維持すること
3. amend 相当保存時に結果 `amended = true` が返ること
4. 同一ユーザでも、保存直前に観測した最新 revision が変化していた場合は無条件で amend しないこと

### 4.2 別ユーザ時の新規 revision

1. 最新 revision の直前更新者が別ユーザである場合、新規 revision が作成されること
2. 新規 revision 作成時に結果 `revision` が増分されること
3. 新規 revision 作成時に結果 `amended = false` が返ること
4. 同一トークンであってもユーザが異なれば amend しないこと

### 4.3 amend 不成立条件

1. 対象が draft の場合に amend 判定へ進まず失敗すること
2. current page を解決できない場合に amend 判定へ進まず失敗すること
3. `content` が空の場合に amend 判定へ進まず失敗すること
4. ロック待機 timeout 時に amend / 新規 revision のいずれにも進まず `Conflict` になること

## 5. 競合制御の観点

### 5.1 最新 revision 競合

1. 追記前に読んだ最新 revision と保存直前の最新 revision が一致している場合に保存できること
2. 保存直前の最新 revision が変化していた場合、即座に古い末尾へ保存しないこと
3. latest revision 競合発生時に、短時間待機と再読込へ進むこと
4. 再読込後に競合が解消していれば、最新本文末尾に対して再度追記を試みること
5. 待機上限を超えても競合が解消しない場合に `Conflict` になること

### 5.2 明示ロック競合

1. 他主体の明示ロックが存在する場合、即保存せず待機へ入ること
2. ロック解消後に保存処理を再開できること
3. ロック待機中に最新 revision も変化した場合、再読込を伴って整合を取り直すこと
4. ロックが待機上限まで解消しない場合に `Conflict` になること

### 5.3 緩和禁止

1. `append` が通常更新より緩い競合制御を取らないこと
2. 人間編集ロックがある場合に、`append` だからという理由で無条件に割り込まないこと
3. latest revision 競合がある場合に、古い本文へそのまま差分を足し込まないこと

## 6. 待機と再試行の観点

### 6.1 待機ループ

1. 待機中に短い間隔で current state を再確認すること
2. 待機ループが無限再試行にならないこと
3. 待機の責務がサービス層にあり、DB 層へ無限再試行を持ち込まないこと
4. 待機解消後に read から保存直前確認までの流れを再実行すること

### 6.2 timeout 時の扱い

1. 待機上限超過時に `Conflict` として返ること
2. timeout 時に部分保存や中途半端な revision 更新が発生しないこと
3. timeout 時に外部エラー区分が `conflict` へ写像されること
4. timeout 時に監査ログ用内部結果分類で競合を識別できること

## 7. 状態別失敗の観点

### 7.1 対象状態

1. current path 解決不能時に `NotFound` になること
2. draft page に対する `append` が失敗すること
3. deleted page を current path として扱わないこと
4. path 不正と対象不存在が別失敗分類になること

### 7.2 競合以外の失敗との分離

1. `InvalidInput` が `Conflict` と混同されないこと
2. `NotFound` が `Conflict` と混同されないこと
3. draft 拒否が `Conflict` と混同されないこと
4. DB 異常が `InternalError` として区別されること

## 8. DB 接続と整合性の観点

### 8.1 保存 API との接続

1. サービス層が current path 解決、権限判定、source 読取、追記後全文生成、amend 判定、競合待機を担当すること
2. DB 層が実際の source 書込、revision 更新、`PageSource` / `PageIndex` 整合維持を担当すること
3. append 実行後に本文、latest revision、更新者情報の整合が保たれること
4. amend 相当保存時に「既存 revision 更新」と「最新者情報更新」が一貫して行われること

### 8.2 再読込後保存

1. 競合解消後の再試行で、古い読み取り結果を再利用せず最新 source を基準に全文を組み立て直すこと
2. 連続競合時に、各周回で最新 revision 前提を更新していること
3. 同時 `append` の整列順序をサーバ内部のアトミック処理へ委ねつつ、最終本文が破損しないこと

## 9. 公開面との接続観点

### 9.1 MCP ツール結果

1. 成功時に `path` と `revision` を返すこと
2. amend 相当保存時に外部へ不要な内部判定詳細を漏らさないこと
3. 競合時に外部エラー区分が `conflict` へ統一されること
4. `append` ツールが amend 指定引数を公開しないこと

### 9.2 既存機能との境界

1. `append` が REST API として公開されないこと
2. MCP からロック取得・更新・解除を直接操作できないこと
3. `append` が `update` と別ツール、別 required scope として扱われること

## 10. 実施レイヤ分割の目安

- 単体テスト向き
  - amend 相当判定
  - 空文字拒否
  - 競合状態からの失敗分類
- サービス層結合テスト向き
  - 最新 revision 再確認
  - ロック待機と再試行
  - 再読込後の全文再構築
  - 同一ユーザ amend と別ユーザ新規 revision の分岐
- DB連動テスト向き
  - amend 時の revision 維持
  - 非 amend 時の revision 増分
  - 本文と index 整合性
- MCP 統合テスト向き
  - `append_page` の応答形
  - `conflict` / `invalid_input` / `not_found` の写像
  - `append` が REST API 非公開であること

## 11. 4.9.2 の整理結果

4.9.2 の完了条件に対する整理結果は以下の通り。

- 同一ユーザ amend の観点を、直前更新者一致、revision 維持、`amended = true` まで含めて定義した
- 別ユーザ時の新規 revision の観点を、revision 増分、`amended = false` まで含めて定義した
- ロック待機の観点を、明示ロック競合、待機、再読込、timeout 時 conflict まで含めて定義した
- 競合失敗の観点を、latest revision 競合、待機上限超過、外部 `conflict` 写像まで含めて定義した
