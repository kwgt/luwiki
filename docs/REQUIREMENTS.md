# 要求仕様

本ドキュメントは、ローカル環境で使用する Wiki システムに関する要求仕様を定義する。
実装支援として Codex CLI 等のツールに入力することを想定し、挙動・制約・優先度を明確化する。

---

## 1. システム概要

- ローカルで利用する個人向け Wiki
- バックエンドは Rust + redb を想定
- ページは Markdown で記述され、HTML にレンダリングして表示される
- ページには履歴（リビジョン）が存在する

---

## 2. ページの基本要件

### 2.1 ページ識別

- 各ページは以下を持つ
  - **page_id**（ULID 等の一意な不変 ID）
  - **path**（人間向けのパス表現）
- page_id はページ作成時に一度だけ発行され、以後変更されない
- path は rename により変更可能

### 2.2 ページパスの仕様
- パスセパレータは'/'
- 絶対パスの場合は'/'から始まること
- UTF-8でエンコード可能な可読文字であれば制限を設けない
- ユーザ入力されたページパスは保存前に正規化し、DBには正規化済みパスのみを保持する
- 正規化済みパスでは、ルートページを除き末尾'/'を認めない


### 2.3 ルートページ

- ルートページのパスは "/" とする
- 初回起動時（最初のユーザ登録後）に雛形データから自動作成する
  - 雛形データは実行ファイルに埋め込みリソースとして保持する
- ルートページは削除不可とする（ソフト／ハードのいずれも禁止）
- ルートページのリネームは禁止とする

### 2.4 パスフォーマット

| パス | 対象 
|:---|:---
| `/wiki/{page_path}` | 閲覧ページ
| `/w/{short_id}` | 短縮URLによる閲覧ページ導線
| `/edit/{page_path}` | 編集ページ
| `/pages/{page_path}` | ページ一覧表示ページ

## 3. ページ履歴（リビジョン）

### 3.1 リビジョン管理

- 各ページは単調増加のリビジョン番号（u64）を持つ
- リビジョンは歯抜けを許可しない
- 最新リビジョンと最古リビジョン（gc_min）はページ単位で管理される

### 3.2 リビジョン操作

- ページ編集は常に新しいリビジョンを作成する
- 例外として、誤字/誤記の修正に限り最新リビジョンを維持する保存（amend）を許可する
- MCP の `append` では、直前更新者と同一ユーザによる直後の追記に限り、amend 相当の保存を許可する
- ページ rename も **1 リビジョンとして扱う**
  - 内容が変わらなくてもリビジョンは増加する

### 3.3 ページ作成・編集の手順ポリシー

- ページ作成および編集の手順は、利用インタフェースの性質に応じて定義できること
- 人間がブラウザUIから直接入力する場合は、段階的な操作手順を採用してよい
  - 例: ドラフト作成、ロック取得、本文編集、保存、キャンセル
- API および MCP のように、クライアントが保存対象の内容を要求時点で提示できるインタフェースでは、単一要求で操作を完結できること
- API および MCP のページ作成・更新系操作は、整合性を単一トランザクションで満たせること
- ブラウザUI向けの段階的な編集手順は、API および MCP にそのまま適用する前提としない
- `ReadOnly` 属性を持つユーザで認証されたブラウザUIでは、書き込み系操作の導線を非活性表示できること
- ブラウザUI上の非活性表示は利便性向上のための補助であり、実際の書き込み禁止はサーバ側の認可で必ず強制すること

### 3.4 ブラウザUIにおける新規ページ作成（ドラフト）

- ブラウザUIからの新規作成はドラフトとして登録する
- ドラフトはソースを持たず、ページIDとパスのみを持つ
- ドラフト作成は `POST /api/pages?path={page_path}` で行い、同時にロックを取得する
- ドラフトに対する `GET /api/pages/{page_id}/source` は 404 を返す
- ドラフトに対する `PUT /api/pages/{page_id}/source` が成功した時点で通常ページとして登録され、リビジョン番号は1となる
- ドラフトはロック解除やロック期限切れにより削除される

## 4. ページ削除

### 4.1 ソフトリムーブ

- ページ削除操作のデフォルトはソフトリムーブ
- ソフトリムーブされたページは以下の扱いとする
  - 一覧には表示されない(検索対象にはなる)
  - データ・履歴は保持される
  - path は予約せず再利用可能
  - 削除済みページは page_id → path の解決を行わない
  - 削除済みページのソースは page_id 指定で取得可能とする

### 4.2 ハードリムーブ

- 明示的な管理操作としてハードリムーブを提供する
- ハードリムーブでは以下を行う
  - 対象ページの全リビジョンを物理削除
  - ページインデックスを削除
- ハードリムーブ後は同一 path の再利用を許可する

#### ドラフト削除

- ドラフトの削除はハードリムーブ扱いとする
- ドラフト削除後は同一 path の再利用を許可する
- ドラフトに紐付くアセットは同時に削除する

---

## 5. ページ rename

### 5.1 基本動作

- rename は page_id を維持したまま path のみを変更する
- rename 操作時に新しいリビジョンを追加する

### 5.2 rename 履歴

- rename を行ったリビジョンには以下の付加情報を持たせる
  - 旧パス（from）
  - 新パス（to）
- rename 履歴は 1 段分のみ保持すればよい

---

## 6. 内部リンク

### 6.1 基本方針

- 内部リンクは以下の二層構造を持つ
  - 入力・表示：path
  - 内部表現：page_id

### 6.2 編集時

- ユーザは相対パス・絶対パスでリンクを記述できる
- 保存時に以下を行う
  - 相対パスを正規化
  - path から page_id を解決
  - 解決できたリンクは page_id 形式に変換して保存

### 6.3 未存在リンク

- リンク先が未存在の場合
  - 未存在リンクとして表示する
  - クリック時に新規ページ作成フローへ遷移する
- エクスポート／インポートに伴うマイグレートで解決不能になったリンクも、未存在リンクと同様に扱う

---

## 7. rename 時のリンクの扱い

### 7.1 他ページからのリンク

- rename 対象ページへの他ページからのリンクは放置する
- page_id により自動的に新しい path へ解決される

### 7.2 rename 対象ページ内のリンク

- 絶対パスリンク：放置
- 相対パスリンク：
  - 可能な限り同一 page_id を指すように変換
  - 変換後に存在状態が変わる場合は警告を通知

---

## 8. リビジョン表示 UI

### 8.1 リビジョン切り替え

- リビジョン番号のドロップダウンで切り替えを行う
- rename リビジョンはドロップダウン内で強調表示する

### 8.2 差分表示（diff）

- 全リビジョンで diff 表示をサポートする
- rename リビジョンも通常の diff 表示を行う
- diff とは別枠で以下を表示する
  - 旧パス → 新パス
  - page_id（同一であることの確認）

---

## 9. 履歴表示

- 履歴表示ボタンを提供する
- 履歴は Markdown で生成する
- 表示時は HTML にレンダリングする
- ダウンロード／クリップボードコピー時は Markdown をそのまま提供する

---

## 10. 補助機能

- 現在ページの page_id を容易に取得できる
- page_id から現在の path を解決できる機能を提供する
  - 削除済みページは対象外とする
- page_id を直接指定したリンク記法を許可する（高度ユーザ向け）

### 10.1 短縮URL

- 各通常ページは `page_id` を基準とした短縮URLを持つ
- 短縮URLは `GET /w/{short_id}` 形式で提供する
- `short_id` は `page_id` と一対一に対応し、同一ページで不変であること
- 短縮URLへのアクセス時は、対応するページの現在の閲覧URL `/wiki/{current_path}` へリダイレクトする
- rename または move 後も、同一 `page_id` を維持する限り同じ短縮URLで到達できること
- 削除済みページは短縮URLの解決対象外とする
- 利用者は現在ページの短縮URLを容易に取得できること
- 利用者は UI または API を通じて短縮URL文字列をコピー利用できること
- `page_id` を直接指定したリンク記法は本文中の高度な参照機能として扱い、短縮URLは閲覧ページへ到達するための共有導線として扱う
- export / import では `page_id` を維持し、短縮URLの対応関係を追加の保存情報なしで再構成できること

## 11. ユーザ認証・ユーザ識別

### 11.1 基本方針

- 本システムはローカル利用を前提とするが、
- 操作主体を識別するためにユーザ認証をサポートする
- 認証の主目的は履歴の帰属とする
- ただし API および MCP では、Bearer スコープおよび path prefix 制約によるアクセス制御も行う

### 11.2 ユーザ識別

- 各ユーザは一意な user_id を持つ
- user_id は以下のいずれかの方式で実現できること
  - ローカル設定ファイルによる固定 ID
  - OS のログインユーザ名を元にした識別子
  - 将来的な拡張として外部認証方式

### 11.3 履歴との関連

- ページの各リビジョンには、その操作を行った user_id を関連付けられること
- 履歴表示時に以下を確認できること
  - 操作日時
  - 操作ユーザ

### 11.4 ブラウザ向け認証

- ブラウザUIからの認証にはBasic認証を用いる

### 11.5 Bearer認証

- APIクライアントおよび将来追加するMCPサーバ機能向けにBearer認証を提供する
- Bearerトークンの発行・失効・削除・一覧表示などの管理操作はCLIでのみ行う
- Bearerトークンの発行対象は登録済みユーザとする
- Bearerトークン種別はアクセス用トークンのみとし、リフレッシュトークンは導入しない
- BearerトークンのデフォルトTTLは30日とする
- Bearerトークンはスライディング期限を採用する
  - 発行時または直近の期限延長時点からTTLの1/2以上が経過した後に認証成功を伴うアクセスが発生した場合、有効期限をその時点からTTL分だけ延長する
  - TTLの1/2未満のアクセスでは有効期限を延長しない
- Bearerトークンの平文はデータベースへ保存しない
  - 照合用のハッシュ値を保存する
  - 管理用の識別子としてtoken_idを別途付与し、その形式はULIDとする
- Bearerトークンは発行時にのみ平文を取得できるものとし、後から再表示できない
- Bearerトークンには少なくとも以下の管理情報を保持する
  - token_id
  - user_id
  - スコープ
  - path prefix 制約
  - 作成日時
  - 最終更新日時
  - TTL
  - 有効期限
  - 失効状態
- Bearerトークンは、対象ユーザが削除された時点で利用不能として扱う
- Bearerトークンには、正規化済みの絶対パスによる path prefix 制約を複数設定できる
- path prefix 制約が設定されている場合、Bearer認証を受け付ける各インタフェースは、要求スコープに加えて操作対象 path が許可された prefix のいずれかに一致することを要求する
- path prefix 制約における `/` は全ページへのアクセスを許可する指定として扱う
- path prefix 制約の判定は、正規化済みパス同士に対して、パス境界を考慮した prefix 判定で行う
- Bearerトークンの path prefix 制約に違反した操作は拒否する
- Bearerトークンの path prefix 制約に違反した操作は監査対象に含める

### 11.6 Bearerトークンのスコープ

- Bearerトークンのスコープは拡張可能な列挙として管理する
- Bearerトークンのスコープは少なくとも `read` 、 `write` 、 `create` 、 `update` 、 `append` 、 `delete` を提供する
- `read` は参照系操作を許可する
- `create` は新規作成系操作を許可する
- `update` は上書き更新および rename を含む更新系操作を許可する
- `append` は末尾追記専用の更新系操作を許可する
- `delete` は削除系操作を許可する
- `write` は後方互換のため維持し、`read` 、 `create` 、 `update` 、 `append` 、 `delete` をまとめて許可する上位互換スコープとして扱う
- `create` 、 `update` 、 `append` 、 `delete` は `read` を暗黙包含しない
- Bearer認証を受け付ける各インタフェースは、要求する最小スコープを定義できること
- Bearerトークンのスコープはユーザ属性による認可制約より優先しない
- `ReadOnly` 属性を持つユーザに紐付く Bearerトークンは、`write` または書き込み系スコープを保持していても、write 系操作には利用できない

### 11.7 ユーザ情報の管理

- ユーザ情報はデータベース上で管理する
- Bearerトークン管理情報はユーザ情報とは別にデータベース上で管理する
- ユーザ情報には、将来拡張可能な属性集合を保持できること
- 初期実装では、Basic認証不可を表す属性 `NoBasicAuth` を提供する
- 初期実装では、write 系操作不可を表す属性 `ReadOnly` を提供する
- `NoBasicAuth` 属性を持たない通常ユーザでは、Basic認証とBearer認証の両方を許可する
- `NoBasicAuth` 属性を持つユーザでは、Basic認証を禁止しBearer認証のみを許可する
- `NoBasicAuth` 属性を持つユーザがBasic認証を試行した場合は認証失敗として扱う
- `ReadOnly` 属性を持つユーザでは、Basic認証 / Bearer認証 / MCP のいずれで認証した場合も write 系操作を禁止する
- `ReadOnly` 属性による write 系操作禁止は Bearerトークンのスコープより優先して適用する
- `ReadOnly` 属性を持つユーザでは、少なくともページ作成、上書き更新、追記、削除、rename、restore、rollback、ロック取得、アセットアップロード、アセット削除を禁止する
- `ReadOnly` 属性による拒否は認証失敗ではなく認可失敗として扱う
- `NoBasicAuth` 属性と `ReadOnly` 属性は独立に付与できる
- ユーザ属性は export/import におけるユーザ情報の一部として保持対象に含める

## 12. MCPサーバ機能

### 12.1 基本方針

- AIアシスタントから利用可能な MCP サーバ機能を提供する
- MCP 機能は、`run` コマンドで明示的に有効化された場合にのみ公開する
- MCP におけるページ操作は path ベースで行い、page_id は公開しない
- MCP におけるページ作成・更新系操作は、クライアントが確定済みの入力内容を要求時点で提示することを前提とする
- MCP におけるページ作成・更新系操作は、ブラウザUI向けのドラフト作成や段階的編集手順を前提とせず、単一要求・単一トランザクションで完結できること
- MCP は Bearer 認証を前提とし、Basic 認証は前提としない
- MCP の認可は Bearer スコープおよび path prefix 制約の両方に従う
- MCP では既存 REST API を内部的に呼び出す構成ではなく、共通処理を利用する専用ハンドラ層を設ける

### 12.2 初期実装で提供する機能

- 初期実装の MCP では、少なくとも以下の機能を提供する
  - ページ取得
  - ページ検索
  - ページ一覧取得
  - ページ作成
  - ページ更新（overwrite）
  - ページ追記（append）
  - ページリネーム
  - セクション取得
- 初期実装の MCP では、削除済みページ参照、restore、アセット操作、ロック操作は提供しない

### 12.3 path ベース操作

- MCP の read 系操作は対象ページの現在 path を基準に認可判定する
- MCP の create は新規作成要求の target path を基準に認可判定する
- MCP の update は対象ページの現在 path を基準に認可判定する
- MCP の append は対象ページの現在 path を基準に認可判定する
- MCP の rename は移動元 path と移動先 path の両方が許可範囲内である場合にのみ許可する
- MCP の検索・一覧で prefix 指定がある場合は、結果の後段フィルタだけでなく要求 prefix 自体も認可判定対象とする
- rename 後の旧 path は自動追跡せず、旧 path 指定時は見つからないものとして扱う

### 12.4 append の扱い

- MCP の `append` は単純な末尾追記に限定する
- `append` はログ用途専用の操作として扱う
- `append` は通常更新より緩い競合制御を許可しない
- `append` の競合時は、必要に応じて一定時間ロック解除を待機し、それでも解除されない場合は競合エラーとして扱う
- `append` は REST API へは公開せず、MCP 内部操作としてのみ提供する

### 12.5 rename / restore 方針

- MCP では rename を提供する
- MCP では restore を提供しない
- MCP の rename は path jail 内同士の場合のみ許可する
- jail 内から jail 外、または jail 外から jail 内への rename は許可しない

## 13. 監査ログ

### 13.1 基本方針

- MCP および関連する認可機能では、操作履歴追跡のための監査ログを記録する
- 監査ログは Wiki ページ本体とは別の専用パスへ保存する
- 監査ログの保存先ストレージは Wiki ページ本体と別物には分離しない

### 13.2 記録対象

- 少なくとも以下の操作は監査ログへ記録する
  - ページ作成
  - ページ更新
  - ページ削除
  - ページリネーム
  - ページ追記（append）
  - 認可失敗
- 認証失敗は監査ログではなく HTTP ログで扱う
- path prefix 制約違反は認可失敗として監査対象に含める

### 13.3 持ち越し事項

- `agent_id` の情報源は設計段階で確定する
- `token_id` の記録形式詳細は設計段階で確定する
- `summary` の生成方針は設計段階で確定する
- `before_revision` / `after_revision` の欠損時表現は設計段階で確定する
- 監査ログの保持期間およびローテーション方針は設計段階で確定する

---
## 14. 全文検索

### 14.1 基本要件

- ページ内容を対象とした全文検索機能を提供する
- 全文検索は以下を満たすこと
  - ページ数増加時にも実用的な性能を維持できる
  - リネームや履歴構造と整合が取れる
  - 英字を含む検索は大文字小文字を区別しない
- 全文検索は独立した検索画面で行い、検索式のリアルタイム評価を行う。

### 14.2 検索対象

- 検索対象は以下の通り
  - 記事中の見出し
  - 記事の本文
  - コードブロック
- 過去リビジョン及びソフトデリート状態の記事を含めて検索対象とする

### 14.3 検索結果

- 検索結果には以下の情報を含める
  - ページ識別子（page_id）
  - ページの現在の path。ただし削除記事の場合は、過去に割り当てられていたパスで代用(その旨が判る表記にする)。
  - マッチしたテキストのスニペット
- 検索結果は入力された検索結果及び、検索対象について最大100件を表示する

### 14.4 履歴・リネームとの関係
- リネームは全文検索の索引単位に影響しない
  - 検索結果は常に 現在の path で表示される
  - 将来的な拡張として、以下を検討可能とする
    - 過去リビジョンを含めた全文検索

---

## 15. アセット管理

- ページに添付されるファイル（画像・資料・その他バイナリ）は アセット と呼ぶ
- アセットはページと紐付いてアップロードされるが、他ページから参照可能である
- アセットには一意な識別子（asset_id、ULID）を付与し、永続的に不変とする
- アセット本体はデータベースに保存せず、ファイルシステムに保存する
  - 保存場所は $XDG_DATA_HOME/<app>/asset/ を基準とする
  - アセットIDに基づき、ディレクトリを2階層（先頭2桁 + 続く3桁）で分散配置する
- ページとアセットの紐付けには多対多関係を許容し、以下を実現する
  - ページ削除時の参照切り離し
  - 他ページからの参照維持
- 内部的には (PageId → AssetId) の対応を管理し、逆引き (AssetId → PageId) は必要に応じて利用する

---

### 15.1 アセット参照記法

- Markdown 内でのアセット参照は asset: プレフィクスを用いる
- 書式例:
```Markdown
![](asset:./foo:banner.png)
```
- asset: 記法には以下を許可する
  - 相対パスによる参照
  - 絶対パスによる参照
  - 所属ページ以外からの参照
- Markdown ソース保存時には asset_id に変換しない<br>→ 保守性・編集性を優先する

---

## 16. テンプレート機能

- ページ編集時に、ページのテンプレートを初期ソースとして用いる機能を実装する。
- テンプレートはシステム設定指定されたWiki上のパスの子ページ(直下の1階層のみ)を用いる
- システム設定でテンプレート格納パスが指定されない場合はテンプレート機能自体を無効化する

---

## 17. ページ一覧表示機能

- 起点ページを指定したページ一覧表示機能を提供する
- ページ一覧表示機能は以下を満たすこと
    - 指定された起点ページ配下のページの一覧を表示する
    - 削除ページも表示し、削除ページの復元のための導線とする
    - ドラフトページは表示対象としない
- ページ一覧表示は100件程度毎にページ切替を行う

## 18. マクロ機能
- マクロ機能は以下の二種類を提供する
    - 入力支援を目的とした即時変換タイプ(日付入力など)。即時変換タイプはソースコードには残らない。
    - 記述拡張または自動化を目的としたレンダリング時変換タイプ(子ページのリスト作成など)。レンダリング時変換タイプはソースコードに残りデータベースにそのまま記録される。
- 特例としてMediaWiki互換の"[[....]]"形式のリンク記述もマクロとして提供する

## 19. エクスポート／インポート

### 19.1 基本方針

- バックアップおよびページツリー単位のマイグレートを目的としたエクスポート／インポート機能を提供する
- エクスポートデータはDB実装に依存しない形式とする
- エクスポートデータは人間可読な構造（JSON/JSONL等）を用いる
- エクスポート時に削除済みページおよび削除済みアセットは含めない

### 19.2 データ種別

- エクスポートデータは `backup` と `migrate` の2種を持つ
- `backup` は全体復元用途とし、インポート時の復元先プレフィクスは `"/"` 固定とする
- `migrate` はサブツリー移送用途とし、インポート時に移送先プレフィクスを指定可能とする

### 19.3 パス表現ルール

- エクスポートデータには基準パス `export_root` を含める
- `export_root` は常に絶対パスで保持する
- `backup` では `export_root` は常に `"/"` とする
- ルートページのパスは特例として `"/"` で保持する
- ページ個別の `path` は `export_root` 基準の相対パスで保持する
- エクスポートデータ中のパス型フィールドには、DBに保持された正規化済みパスを用いる
- パス型フィールドでは `..` を含まない最短表現を用いる
- `migrate` インポート時の最終配置パスは、`export_root` 基準の相対パスを移送先プレフィクスへ連結し、正規化して決定する
- ルートページを対象とする `migrate` は禁止する
- `revisions.jsonl` に格納されるMarkdownソース中のリンク文字列は、ユーザ記述をそのまま保持し、パス型フィールドの正規化規則は適用しない

### 19.4 リネーム情報の扱い

- `backup` ではリネーム関連情報（ページ単位・リビジョン単位）を保持する
- `migrate` ではリネーム関連情報を保持しない
- `migrate` データにリネーム情報が含まれていた場合は、通常モードでは無視して警告し、厳格モードではエラーとして中断する
- `backup` に含まれる `rename.from` / `rename.to` は、正規化済みの絶対パスで保持する

### 19.5 ユーザ情報の扱い

- `backup` ではユーザ情報を認証情報を含めて復元対象に含める
- `migrate` でもユーザ情報は `UserInfo` 構造体のオブジェクトとして追加対象に含める
- `migrate` では `--user-map` 未指定ユーザも追加対象に含める
- ユーザ属性はユーザ情報の一部として export/import 対象に含める
- インポート完了後に `username` が重複する状態になる場合は、`strict-mode` の有無に関わらずエラーとして中断する
- Bearerトークンはエクスポート／インポート対象に含めない
- Bearerトークンが必要な場合は、インポート後にCLIから再発行する

### 19.6 マイグレート時の実行保証

- エクスポートおよびインポートはCLIコマンドでのみ実行し、実行中はサーバ停止状態を前提とする
- `migrate` 用のエクスポートは、エクスポート処理と移送元ページツリー削除を単一トランザクションで行う
- `migrate` 用のエクスポートが成功した時点で、移送元ページツリーは無条件に削除する
- ドラフトページはエクスポート対象に含めないが、対象ツリー削除時には削除対象に含める
- ロック中のページはエクスポート対象に含める
- 移送元ページツリーの削除時に、対象ページに紐付くロック情報も削除する
- `migrate` 用のエクスポートまたはインポートで失敗が発生した場合はロールバックする

### 19.7 リハーサルモード

- `backup` の `dry-run` では追加の検証は行わない
- `migrate` の `dry-run` では、少なくとも以下を検証対象に含める
  - ツリー外へのページリンクが含まれていないこと
  - ツリー外へのアセットリンクが含まれていないこと
  - インポート完了後に `username` の重複が発生しないこと
  - 無効リンクへの置換対象となるエントリの有無
  - 移送先に子ページを持つ既存パスが存在しないこと

### 19.8 インポート時の整合性検証

- `migrate` インポート時は、移送先に子ページを持つ既存パスが存在する場合はエラーとする
- インポート時はエクスポートデータ内の参照整合性を検証する
- `manifest` に記録された件数は、対応するJSONLの実件数と一致していなければならない
- アセットについては、`assets.jsonl` の件数と実体ファイル数の一致も検証対象に含める
- エクスポートデータ中のID、およびインポート先の同種IDとの重複を検出した場合はエラーとして中断する
- JSONL間で参照切れがある場合は異常データとして中断する
- 参照整合性検証には、少なくともJSONL間の参照切れ、アセット実体ファイルの欠落、およびアセットサイズ不一致検出を含める

### 19.10 エクスポート対象のページ状態・アセット状態

- ロック情報自体はエクスポート対象に含めない
- ロック情報は多重編集抑止のための一時情報として扱い、ページの凍結状態とはみなさない
- 削除済みアセットはエクスポートデータに含めない
- アセットは編集不能オブジェクトとして扱う
- 孤立アセットは、ページパスを軸としたエクスポート対象選定の結果、対象外とする

### 19.11 暗号化ZIPの扱い

- エクスポートデータ暗号化時は、利用可能であれば AES-256 を優先して使用する
- AES-256 が利用できない場合は、Standard ZIP 2.0 へフォールバックしてよい
- Standard ZIP 2.0 へフォールバックした場合は警告を表示する
- パスワード仕様は採用したZIP暗号方式に従う
- 復号失敗、不正パスワード、未対応形式の受信はすべてエラーとして扱う
- 暗号方式情報はZIPレイヤーの責務とし、エクスポートデータ本体には保持しない

### 19.9 マイグレート時のリンクの扱い

- `migrate` の `strict-mode` では、少なくとも以下をエラー対象とする
  - ツリー外のページへのリンク
  - 絶対パスによるページリンク
- 非`strict-mode`では、上記リンクは移送先で未解決リンクとして扱う
- `--fix-broken-link` が指定されている場合、未解決リンクは `about:invalid` へ置換する
- `--fix-broken-link` が指定されていない場合、未解決リンクはそのまま保持する
- 削除済みページへのリンクは、通常の未存在リンクと同様に扱う
- `page_id` 直接指定リンクについては、マイグレート仕様上の考慮対象外とする
- `migrate` ではrename情報を用いた再解決は行わない
