# Bearer認証 テスト観点一覧

本書は、`docs/BEARER_AUTH_DESIGN.md` の 5.6 で整理した主要テスト観点を、
実装・レビュー・テストケース分解に利用できる粒度へ展開した別紙である。

Bearer認証は Basic認証との共通入口、Bearerトークン管理情報、スコープ判定、
スライディング期限、CLI管理操作、ユーザ管理連動が相互に関係するため、
本書ではそれらを責務単位で整理する。

## 2. 観点の使い方

- 本書はテストケースのたたき台であり、個々の観点を単体テスト、結合テスト、CLIテストへ配分して具体化する
- 全観点を同一レイヤで検証する前提にはしない
- ステータスコード、レスポンスヘッダ、DB更新、副作用有無のどれを確認対象にするかを各観点で明示する
- `token list` の表示仕様のような暫定事項は、表示体裁の固定ではなく誤認しないことを優先して評価する

## 3. 認証入口の観点

### 3.1 `Authorization` ヘッダ件数と形式

1. `Authorization` ヘッダが存在しない場合、401 Unauthorized を返すこと
2. `Authorization` ヘッダが複数存在する場合、400 Bad Request を返すこと
3. `Authorization` ヘッダが1件で `Basic` の場合、Basic認証へ分岐すること
4. `Authorization` ヘッダが1件で `Bearer` の場合、Bearer認証へ分岐すること
5. 未対応 scheme の場合、400 Bad Request を返すこと
6. scheme 名は正しいが資格情報部が欠落している場合、400 Bad Request を返すこと
7. 不正なヘッダ形式を後続ハンドラへ渡さないこと

### 3.2 Basic認証の観点

1. 正しいユーザ名・パスワードで認証成功し、認証文脈へユーザ情報が格納されること
2. 誤ったパスワードで 401 Unauthorized になること
3. 存在しないユーザ名で 401 Unauthorized になること
4. Basic認証時は全スコープ相当として認可判定を通過できること
5. Basic認証時は `X-Bearer-Expire` を付与しないこと

### 3.3 Bearer認証の観点

1. 正しい Bearerトークンで認証成功し、認証文脈へユーザ情報とスコープ集合が格納されること
2. 未発行トークンで 401 Unauthorized になること
3. 照合失敗トークンで 401 Unauthorized になること
4. 失効済みトークンで 401 Unauthorized になること
5. 期限切れトークンで 401 Unauthorized になること
6. トークンに紐付く `user_id` の対象ユーザが存在しない場合、401 Unauthorized になること
7. Bearer認証成功時でも `token_id` やトークン平文が認証文脈へ露出しないこと

## 4. 認可・スコープ判定の観点

### 4.1 スコープ集合の観点

1. `read` スコープのみを持つトークンが `read` 要求を満たすこと
2. `write` スコープのみを持つトークンが `read` 要求を満たすこと
3. `write` スコープのみを持つトークンが `write` 要求を満たすこと
4. `read` スコープのみを持つトークンが `write` 要求を満たさず 403 Forbidden になること
5. 複数スコープ指定時に重複があっても集合として正しく扱えること

### 4.2 API要件との対応

1. `read` 要求APIで Bearer `read` が通ること
2. `read` 要求APIで Bearer `write` が通ること
3. `write` 要求APIで Bearer `write` が通ること
4. `write` 要求APIで Bearer `read` が 403 Forbidden になること
5. `GET /api/users/me` が Bearer `read` で利用できること
6. OpenAPI の `x-required-scope` と実装上の必要スコープ指定が一致すること

### 4.3 ロック認証との順序

1. Bearer認証不成立時は `X-Lock-Authentication` より前に 401 Unauthorized になること
2. Bearer認証成功だが必要スコープ不足の場合は `X-Lock-Authentication` より前に 403 Forbidden になること
3. Bearer認証・スコープ判定を通過した後にのみロック認証を評価すること
4. ロック条件違反時に 403 Forbidden または 423 Locked の責務分担が崩れないこと

## 5. スライディング期限の観点

### 5.1 延長判定

1. `ttl / 2` 未満の経過では延長しないこと
2. `ttl / 2` ちょうどで延長すること
3. `ttl / 2` を超えた場合に延長すること
4. 期限切れ後のアクセスは延長せず 401 Unauthorized になること
5. Bearer認証失敗時は延長判定を行わないこと

### 5.2 更新内容

1. 延長時に `expire_at = now + ttl` へ更新されること
2. 延長時に `updated_at` が同じ `now` で更新されること
3. 延長不要時は `expire_at` と `updated_at` を更新しないこと
4. 更新は主テーブルに対する単一の整合した書き込みとして扱われること

### 5.3 レスポンスヘッダ

1. 延長が発生した Bearer認証成功時のみ `X-Bearer-Expire` が付与されること
2. `X-Bearer-Expire` には更新後 `expire_at` が ISO8601 のタイムゾーン無し表記で設定されること
3. 延長不要な Bearer認証成功時は `X-Bearer-Expire` を付与しないこと
4. Basic認証時は `X-Bearer-Expire` を付与しないこと

## 6. データモデル・永続化の観点

### 6.1 Bearerトークン保存

1. Bearerトークン平文をDBへ保存しないこと
2. 照合用ハッシュ値をキーとして主テーブルへ保存できること
3. `token_id` 変換テーブルから主テーブルへ到達できること
4. Bearerトークン管理情報に `token_id` / `user_id` / `scopes` / `created_at` / `updated_at` / `ttl` / `expire_at` / `revoked` / `name` が保持されること
5. `last_used_at` を保持しないこと

### 6.2 トランザクション整合

1. `token create` で主テーブルと `token_id` 変換テーブルが整合して作成されること
2. TTL延長で主テーブル更新結果が不整合なく反映されること
3. `token purge` で主テーブルと `token_id` 変換テーブルが同一処理単位で削除されること
4. ユーザ削除時に関連トークンと `token_id` 変換情報が同一処理単位で削除されること

## 7. CLI管理操作の観点

### 7.1 `token create`

1. 正常時に Bearerトークン管理情報が保存されること
2. 正常時に平文トークンが一度だけ出力されること
3. `token_id` が ULID 形式であること
4. 指定スコープが `BearerScopeSet` として保存されること
5. 指定TTLが `ttl` と `expire_at` へ正しく反映されること
6. 任意名がある場合は `name` に保存されること
7. 不正な `--scope` でエラーになること
8. 不正な `--ttl` または 0 以下の `--ttl` でエラーになること
9. 存在しないユーザ指定でエラーになること

### 7.2 `token revoke`

1. 有効トークンを失効状態へ変更できること
2. 失効時に `updated_at` が更新されること
3. 既失効トークンを対象にしてもエラーにせず警告扱いできること
4. 期限切れトークンを対象にしてもエラーにせず警告扱いできること
5. `TOKEN-ID` / `--user` / `--all` の指定制約違反でエラーになること
6. 存在しない `TOKEN-ID` 指定でエラーになること
7. 存在しないユーザ指定でエラーになること

### 7.3 `token purge`

1. `TOKEN-ID` 指定で1件削除できること
2. `--expired` 指定で期限切れトークンを削除対象に含めること
3. `--revoked` 指定で失効済みトークンを削除対象に含めること
4. `--expired` と `--revoked` 併用時に和集合を対象とすること
5. 削除時に主テーブルと `token_id` 変換テーブルがともに削除されること
6. 指定制約違反でエラーになること
7. 存在しない `TOKEN-ID` 指定でエラーになること

### 7.4 `token list`

1. 一覧表示でトークン平文を表示しないこと
2. 一覧表示で `token_id`、対象ユーザ名、有効期限が表示できること
3. 詳細表示で `created_at`、`updated_at`、`revoked`、`name` が表示できること
4. `user_id` から最新ユーザ名を解決して表示すること
5. `--expired` の判定がコマンド実行時点の現在時刻で行われること
6. `--revoked` と `--expired` 併用時に和集合条件で抽出されること
7. 暫定の状態表示欄が少なくともスコープと失効状態を判別可能にしていること
8. `updated_at` が最終使用日時ではなく最終更新日時であることが誤認されにくいこと

## 8. ユーザ管理連動の観点

1. ユーザ削除時に関連する Bearerトークン管理情報が削除されること
2. ユーザ削除後に対象トークンで認証できないこと
3. ユーザ削除後に `token list` へ残存表示されないこと
4. ユーザ名変更後に `token list` が最新ユーザ名を表示すること
5. Bearerトークン管理情報が `UserInfo` に内包されていないこと

## 9. エラー応答・ログの観点

### 9.1 エラー応答

1. 400 / 401 / 403 / 423 の責務分担が設計どおりであること
2. 認証失敗時のレスポンスが JSON の `reason` フィールドを持つこと
3. 認可失敗時のレスポンスが認証失敗と混同されないこと
4. Bearer認証失敗が後続ハンドラへ委譲されないこと

### 9.2 ログ

1. Bearerトークン平文をログへ出力しないこと
2. Basic認証の Base64 展開後資格情報をログへ出力しないこと
3. `token_id` が特定できる場合に限って識別情報として利用すること
4. 失敗理由の粒度が未発行、照合失敗、失効済み、期限切れ、ユーザ未解決などに留まること
5. スコープ不足による 403 を認証失敗ログへ混在させないこと

## 10. 実施レイヤ分割の目安

- 単体テスト向き
  - スコープ集合の包含判定
  - TTL延長判定
  - 状態導出
  - CLI引数制約
- 結合テスト向き
  - 認証入口から認証文脈格納まで
  - `X-Bearer-Expire` 付与
  - ステータスコード責務分担
  - ロック認証との順序
- DB連動テスト向き
  - 主テーブルと `token_id` 変換テーブルの整合
  - ユーザ削除連動
  - CLI操作による状態遷移

## 11. 未確定事項との関係

- `token list` の状態表示欄の短縮表現は暫定事項であるため、本書では「誤認しないこと」を主眼に観点化する
- 詳細な表示体裁そのものは、使用感評価の結果を受けて別途固定する
- 本書で不足が見つかった場合は、`docs/BEARER_AUTH_DESIGN.md` または関連仕様への補足要否を 5.5 タスクで再判定する
